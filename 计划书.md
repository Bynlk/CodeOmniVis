## 三、P0：N+1 查询静态检测

### 3.1 问题描述

```typescript
// ❌ 循环内的 DB 查询 = N+1
const users = await prisma.user.findMany()
for (const user of users) {
  const bookings = await prisma.booking.findMany({
    where: { userId: user.id }
  })
}
```

这是后端性能最常见的隐患之一，目前没有任何工具做静态检测。

### 3.2 实现方案

**新建文件:** `packages/analyzer/src/resolver/nPlusOneDetector.ts`

**核心算法:**
```
输入: handler/service 节点列表 + SymbolResolver
输出: Issue[]

对每个节点:
  1. 用 ts-morph 打开源文件，定位函数
  2. 遍历函数体 AST，查找循环结构:
     - ForStatement / ForOfStatement / ForInStatement
     - CallExpression(.forEach) / CallExpression(.map) / CallExpression(.flatMap)
  3. 对每个循环体:
     a. 收集所有 CallExpression
     b. 用 extractDbCall() 判断是否是 DB 调用
     c. 如果是 → 生成 Issue
  4. 额外检查: await 在循环体内（非 Promise.all 包装）
```

**AST 遍历模式:**
```typescript
import { SyntaxKind, Node, ForStatement, ForOfStatement, CallExpression } from 'ts-morph'

// 查找循环
const forLoops = fn.getDescendantsOfKind(SyntaxKind.ForOfStatement)
const forEachCalls = fn.getDescendantsOfKind(SyntaxKind.CallExpression)
  .filter(call => {
    const methodName = call.getExpression().getLastChild()?.getText()
    return ['forEach', 'map', 'flatMap'].includes(methodName)
  })

// 对每个循环，检查内部的 DB 调用
for (const loop of forLoops) {
  const dbCalls = loop.getDescendantsOfKind(SyntaxKind.CallExpression)
    .filter(call => this.extractDbCall(call))
  if (dbCalls.length > 0) {
    issues.push(this.createNPlusOneIssue(loop, dbCalls, handlerNode))
  }
}
```

**输出 Issue 格式:**
```typescript
{
  id: `n1-${handlerNode.id}-${dbCall.line}`,
  type: 'n_plus_one_query',  // 新增 IssueType
  severity: 'critical',
  description: `N+1 query: prisma.${dbCall.modelName}.${dbCall.operation}() inside ${loopType} loop`,
  locations: [
    { file: handlerNode.filePath, line: loopLine, note: 'loop starts here' },
    { file: handlerNode.filePath, line: dbCall.line, note: 'DB query inside loop' }
  ],
  relatedNodeIds: [handlerNode.id, dbModelNode?.id],
  relatedEdgeIds: []
}
```

### 3.3 集成点

在 `CrossLayerLinker.link()` 末尾调用:
```typescript
// 在 link() 方法的 return 之前
const nPlusOneIssues = await this.detectNPlusOne(graph, nodeMap)
// 将 issues 附加到 result
```

## 四、P0：Auth 覆盖检测

### 4.1 问题描述

```typescript
// ❌ 没有鉴权的 API 路由
export async function POST(req: Request) {
  return prisma.user.findMany()  // 未鉴权就读数据库
}
```

### 4.2 实现方案

**新建文件:** `packages/analyzer/src/resolver/authDetector.ts`

**Auth 函数白名单:**
```typescript
const AUTH_PATTERNS = [
  // NextAuth / Next.js
  'getServerSession', 'getSession', 'auth',
  // JWT / Token
  'verifyToken', 'getToken', 'jwtVerify',
  // 通用
  'checkAuth', 'requireAuth', 'isAuthenticated', 'withAuth',
  // NestJS
  'useGuards', 'canActivate',
  // Express middleware
  'passport.authenticate', 'verify',
]
```

**算法:**
```
对每个 api_route/handler 节点:
  1. 打开源文件，定位 handler 函数
  2. 收集函数体内所有 CallExpression 的函数名（递归）
  3. 检查 import 语句中是否导入了 auth 函数
  4. 如果函数体内没有调用任何白名单函数:
     a. 检查是否是公开路由（health, webhook 等）
     b. 如果不是 → Issue(type: 'unguarded_route', severity: 'critical')
```

**特殊处理:**
- 跳过 GET 请求的公开路由模式: `/api/health`, `/api/webhook/*`
- 检查 middleware 级别 auth: 扫描 `middleware.ts` 中的 `matcher` 配置
- 检查 tRPC middleware: `publicProcedure` vs `protectedProcedure`

### 4.3 输出格式

```typescript
{
  id: `auth-${routeNode.id}`,
  type: 'unguarded_route',
  severity: 'critical',
  description: `API route ${route} has no authentication guard`,
  locations: [
    { file: routeNode.filePath, line: routeNode.line, note: 'route definition' }
  ],
  relatedNodeIds: [routeNode.id, handlerNode?.id],
  relatedEdgeIds: []
}
```

### 4.4 需要修改的文件

| 文件 | 操作 |
|------|------|
| `resolver/authDetector.ts` | 新建 |
| `shared/types/issue.ts` | 添加 `unguarded_route` 到 IssueType |

## 五、P1：React Server Component 边界分析

### 5.1 问题描述

Next.js App Router 的核心心智模型:
- `'use client'` 组件可以使用 useState/useEffect
- `'use server'` 函数只能在服务端执行
- Server Component 不能 import Client Component（反向可以）

目前没有任何工具可视化这个边界。

### 5.2 实现方案

**新建文件:** `packages/analyzer/src/resolver/rscBoundaryDetector.ts`

**修改文件:** `parsers/reactComponent.ts`, `parsers/nextjsApp.ts`

**步骤 1: 检测指令**
在 ReactComponentParser 和 NextjsAppParser 中，解析文件时检查头部:
```typescript
private detectDirective(content: string): 'client' | 'server' | 'none' {
  const trimmed = content.trimStart()
  if (trimmed.startsWith("'use client'") || trimmed.startsWith('"use client"')) return 'client'
  if (trimmed.startsWith("'use server'") || trimmed.startsWith('"use server"')) return 'server'
  return 'none'
}
```

**步骤 2: 扩展 ComponentMetadata**
```typescript
// shared/types/node.ts
interface ComponentMetadata {
  props: string[]
  hasState: boolean
  isPage: boolean
  jsxChildCount: number
  directive?: 'client' | 'server' | 'none'  // 新增
}
```

**步骤 3: 边界违规检测**
```
对每个 server component:
  检查其 import 列表中是否有 'use client' 组件
  如果有 → Issue(type: 'rsc_boundary_violation', severity: 'warning')
```

### 5.3 输出格式

```typescript
{
  id: `rsc-${serverComponent.id}`,
  type: 'rsc_boundary_violation',
  severity: 'warning',
  description: `Server Component "${name}" imports Client Component "${clientName}"`,
  locations: [
    { file: serverComponent.filePath, line: importLine, note: 'import of client component' }
  ],
  relatedNodeIds: [serverComponent.id, clientComponent.id],
  relatedEdgeIds: []
}
```

### 5.4 需要修改的文件

| 文件 | 操作 |
|------|------|
| `resolver/rscBoundaryDetector.ts` | 新建 |
| `parsers/reactComponent.ts` | 解析时检测 'use client' 指令 |
| `parsers/nextjsApp.ts` | 解析时检测 'use client'/'use server' 指令 |
| `shared/types/node.ts` | ComponentMetadata 添加 directive 字段 |
| `shared/types/issue.ts` | 添加 `rsc_boundary_violation` 到 IssueType |

## 六、P2：动态 API 调用识别

### 6.1 问题描述

当前 ApiCallsParser 只能识别静态 URL:
```typescript
fetch('/api/booking')  // ✅ 能识别
fetch(`/api/booking/${id}`)  // ❌ 识别不了
const url = '/api/booking'; fetch(url)  // ❌ 识别不了
```

### 6.2 实现方案

**修改文件:** `parsers/apiCalls.ts`

在 `extractApiCalls()` 方法中添加常量折叠:
```typescript
// 简单常量折叠：追踪变量赋值
const varMap = new Map<string, string>()
sourceFile.forEachDescendant(node => {
  if (Node.isVariableDeclaration(node)) {
    const init = node.getInitializer()
    if (init && Node.isStringLiteral(init)) {
      varMap.set(node.getName(), init.getLiteralValue())
    }
  }
})

// 模板字符串处理
if (Node.isTemplateExpression(urlNode)) {
  const head = urlNode.getHead().getText()
  const parts = urlNode.getTemplateSpans().map(span => {
    const expr = span.getExpression()
    // 尝试从 varMap 解析
    return varMap.get(expr.getText()) || '${...}'
  })
  // 构建模式: /api/booking/${...}
}
```

## 七、P2：Prisma 别名追踪

### 7.1 问题描述

```typescript
const db = prisma  // 别名
const result = await db.user.findMany()  // 当前可能识别不了
```

### 7.2 实现方案

**修改文件:** `resolver/symbolResolver.ts`

在 `extractDbCall()` 中添加别名追踪:
```typescript
// 在 traceFunction 开始时，收集 prisma 别名
const prismaAliases = new Set<string>()
prismaAliases.add('prisma')

const assignments = fn.getDescendantsOfKind(SyntaxKind.VariableDeclaration)
for (const decl of assignments) {
  const init = decl.getInitializer()
  if (init && prismaAliases.has(init.getText())) {
    prismaAliases.add(decl.getName())
  }
}

// 在 extractDbCall 中使用 prismaAliases
const expr = callExpr.getExpression().getText()
const parts = expr.split('.')
if (prismaAliases.has(parts[0])) {
  // 匹配 prisma 模式
}
```

## 八、集成到分析流程

### 8.1 修改 CrossLayerLinker.link()

```typescript
async link(graph: OmniGraph): Promise<CrossLayerResult> {
  // ... 现有的 5 个阶段 ...

  // Phase 6: 深度分析检测器
  const issues: Issue[] = []

  // N+1 检测
  const nPlusOneDetector = new NPlusOneDetector(this.symbolResolver)
  issues.push(...nPlusOneDetector.detect(graph))

  // Auth 覆盖检测
  const authDetector = new AuthDetector()
  issues.push(...authDetector.detect(graph))

  // RSC 边界检测
  const rscDetector = new RSCBoundaryDetector()
  issues.push(...rscDetector.detect(graph))

  return {
    edges,
    stats: { ... },
    issues,  // 新增
  }
}
```
