## 1. 项目定位与动机

### 1.1 一句话定义

CodeOmniVis 是一个 **零配置 CLI 工具**，针对 TypeScript 全栈项目（Next.js + tRPC/Express + Prisma/TypeORM），自动提取三层结构关系并生成可交互的拓扑图：

```
前端页面 → 组件树 → API 调用
                          ↕
              后端路由 → Handler → Service
                                        ↕
                              数据库表 → 关系图
```

### 1.2 核心动机

- 为 GitHub 开源项目积累流量与影响力，服务求职
- 填补市场空白：现有工具要么只看前端、要么只看后端、要么只看数据库，没有一张图把三层连起来
- 构建 AI 原生工具：通过 MCP 协议让 Cursor / Claude Desktop 能"看懂"整个系统

### 1.3 目标用户

**主要用户**：加入新项目需要快速上手的全栈工程师

**次要用户**：
- AI 编码助手（通过 MCP 接口消费数据）
- 需要向团队讲解系统架构的 Tech Lead
- 开源项目贡献者（快速理解陌生 codebase）

## 2. 竞品分析（诚实版）

| 工具 | 做了什么 | 真正的弱点 |
|------|---------|-----------|
| **CodeLogic** | 静态+运行时依赖分析，已有 MCP 接口 | 闭源、企业收费、只支持 Java/.NET、无前端组件层、无全栈视图 |
| **dependency-cruiser** | 文件级依赖图 | 无语义区分（不懂 API vs 组件 vs DB），无跨层连线 |
| **Sourcetrail** | 代码符号关系图 | 已停止维护（archived 2021），无 Web 框架语义 |
| **Swagger / OpenAPI** | API 文档与定义 | 只显示 API 定义，需手动维护，无前端、无 DB |
| **Storybook** | UI 组件展示 | 只显示组件，无后端、无逻辑依赖 |
| **Eraser.io** | AI 辅助架构图绘制 | 手动绘制 / AI 生成，非从代码自动提取 |
| **XState Viz** | 状态机可视化 | 只针对 XState，不整合系统架构 |

**CodeOmniVis 的真实差异点**：
1. 唯一同时覆盖前端组件树、后端 API 链、数据库 ER 的开源工具
2. 针对 TypeScript 生态（Next.js + tRPC + Prisma）深度优化，非通用静态分析
3. MCP 原生：不是事后接入，而是从设计之初就作为一等公民的输出格式
4. 开源 MIT + npx 零安装门槛

## 3. 差异化定位

### 3.1 不是什么

- ❌ 不是代码质量检查工具（无 lint、无 bug 检测）
- ❌ 不是性能监控工具（无运行时指标）
- ❌ 不是文档生成工具（无需手动维护）
- ❌ 不是通用多语言工具（MVP 专注 TypeScript）

### 3.2 是什么

- ✅ 是一个**架构认知工具**：让开发者在 60 秒内看懂陌生系统
- ✅ 是一个**AI 上下文工具**：让 LLM 不再盲目修改代码
- ✅ 是一个**一致性检查工具**：发现前端调用了后端不存在的 API

## 4. MVP 功能边界

### 4.1 MVP 包含（v1.0）

| 功能 | 说明 |
|------|------|
| Prisma schema 解析 → ER 图 | 提取所有 Model、Field、Relation，生成数据库关系图 |
| TypeORM entity 解析 → ER 图 | 提取 `@Entity`、`@Column`、`@OneToMany` 等装饰器 |
| Next.js App Router 路由提取 | 扫描 `app/` 目录结构，提取所有页面路由 |
| Next.js Pages Router 路由提取 | 扫描 `pages/` 目录，兼容旧项目 |
| Express 路由提取 | 提取 `app.get/post/put/delete` 和 `router.xxx` 定义 |
| tRPC router 解析 | 提取 `createTRPCRouter`、`procedure` 定义及其 resolver |
| React 组件树分析 | 解析 JSX/TSX import 关系，构建组件依赖树 |
| 前端 API 调用提取 | 识别 `fetch()`、`axios()`、`trpc.xxx.useQuery/useMutation` 调用 |
| 跨层连线 | 将前端 API 调用与后端 handler 连接，handler 与 DB 操作连接 |
| 分层聚合视图 | 按路由前缀/模块折叠节点，默认显示 6-10 个顶层模块 |
| 前后端接口一致性检测 | 报告死链（前端调用了不存在的 API）和冗余路由 |
| MCP Server | 提供 3 个工具：`get_api_routes`、`get_component_tree`、`find_callers` |
| CLI：`npx codeomnivis serve` | 启动本地 Web 可视化服务 |
| CLI：`npx codeomnivis analyze` | 生成静态 JSON 数据文件 |
| TypeScript path alias 解析 | 支持 `tsconfig.json` 中的 `paths` 配置（如 `@/components/...`） |
| monorepo 支持 | 自动检测 Turborepo / pnpm workspace，识别包边界 |

### 4.2 MVP 不包含（移至 v2）

| 功能 | 原因 |
|------|------|
| XState 状态机分析 | cal.com demo 项目不使用 XState，会产生空白输出 |
| 基础设施层（K8s/Docker） | cal.com 的 K8s 配置不在公开仓库 |
| OpenTelemetry 运行时追踪 | 大幅增加部署复杂度，MVP 纯静态分析即可 |
| 消息队列流（Kafka/RabbitMQ） | 大多数 Web 应用不使用，降低 demo 普遍性 |
| 自然语言查询 | 增加不稳定性，不是核心差异点 |
| VS Code 扩展 | CLI 先行，扩展作为 v1.5 |
| Python/Go 后端支持 | MVP 专注 TypeScript 生态，确保深度 |
| Vue/Angular 前端支持 | 同上 |
| CI/CD 集成 | 作为 v2 杀手功能 |

### 4.3 配置文件（`.codeomnivis.json`）

"零配置"意味着无需配置也能运行，但支持可选配置文件覆盖默认行为：

```json
{
  "root": ".",
  "frontend": {
    "dirs": ["apps/web/src", "packages/ui/src"],
    "framework": "next"
  },
  "backend": {
    "dirs": ["apps/api/src", "apps/server/src"],
    "framework": "express"
  },
  "database": {
    "prismaSchema": "packages/db/prisma/schema.prisma",
    "typeormDirs": ["src/entities"]
  },
  "exclude": ["node_modules", "dist", ".next", "coverage"],
  "port": 4321
}
```

**自动检测逻辑**（无配置文件时）：
1. 读取根目录及所有子包的 `package.json`
2. 依赖中含 `next` → 标记为 Next.js 前端
3. 依赖中含 `express` → 标记为 Express 后端
4. 依赖中含 `@trpc/server` → 标记为 tRPC 项目
5. 存在 `prisma/schema.prisma` → 标记为 Prisma 项目
6. 存在 `tsconfig.json` → 读取 `paths` 配置

## 6. 核心数据模型

所有解析结果统一转换为以下图模型，存储于 SQLite 并通过 REST API 暴露。

### 6.1 节点类型（NodeType）

```typescript
type NodeType =
  | 'page'           // Next.js 页面路由（/app/booking/page.tsx → /booking）
  | 'component'      // React 组件
  | 'api_route'      // Next.js API Route（/app/api/booking/route.ts）
  | 'trpc_procedure' // tRPC procedure（booking.create）
  | 'express_route'  // Express 路由（POST /api/booking）
  | 'handler'        // 路由 handler 函数（在调用链中作为中间节点）
  | 'service'        // Service 层函数/类方法（Handler → Service 调用链）
  | 'db_model'       // Prisma Model / TypeORM Entity（DB 层主节点）
  | 'module'         // 聚合节点（代表一组路由/组件的折叠视图）
  // 注：db_field 字段信息存储在 db_model.metadata.fields 数组中，
  // 不作为独立节点，避免节点数量爆炸（单个 model 可能有 30+ 字段）
```

### 6.2 边类型（EdgeType）

```typescript
type EdgeType =
  | 'renders'        // 组件渲染关系：Parent → Child
  | 'navigates_to'   // 页面导航：Page A → Page B（通过 Link/router.push）
  | 'calls_api'      // 前端 API 调用：Component → API Route / tRPC Procedure
  | 'handles'        // 路由处理：API Route → Handler Function
  | 'calls_service'  // Handler → Service 调用
  | 'queries_db'     // Service/Handler → DB Model 查询
  | 'db_relation'    // DB 关系：Model A → Model B（一对多/多对多）
  | 'imports'        // 模块导入（用于解析但通常不直接显示）
  | 'contains'       // 聚合关系：Module → 其内部节点
```

### 6.3 节点数据结构

```typescript
interface OmniNode {
  id: string              // 唯一ID，格式：{type}:{filePath}:{name}
  type: NodeType
  name: string            // 显示名称
  filePath: string        // 相对于项目根目录的路径
  line: number            // 定义所在行号
  column: number          // 定义所在列号
  metadata: Record<string, unknown>  // 类型特定的额外信息
  // 类型特定 metadata 示例：
  // page: { route: string, isDynamic: boolean, params: string[] }
  // api_route: { method: string, route: string }
  // trpc_procedure: { type: 'query'|'mutation'|'subscription', router: string }
  // db_model: { tableName: string, fieldCount: number }
  // component: { props: string[], hasState: boolean }
}
```

### 6.4 边数据结构

```typescript
interface OmniEdge {
  id: string              // 唯一ID，格式：{sourceId}--{type}--{targetId}
  source: string          // 源节点 ID
  target: string          // 目标节点 ID
  type: EdgeType
  confidence: 'certain' | 'inferred'
  // certain: 通过直接 import 或字符串匹配确认
  // inferred: 通过模式匹配推断（如动态路由）
  metadata: Record<string, unknown>
}
```

### 6.5 SQLite Schema

```sql
CREATE TABLE nodes (
  id          TEXT PRIMARY KEY,
  type        TEXT NOT NULL,
  name        TEXT NOT NULL,
  file_path   TEXT NOT NULL,
  line        INTEGER NOT NULL,
  col         INTEGER NOT NULL,
  metadata    TEXT NOT NULL DEFAULT '{}',  -- JSON string
  updated_at  INTEGER NOT NULL             -- Unix timestamp ms
);

CREATE TABLE edges (
  id          TEXT PRIMARY KEY,
  source      TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  target      TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  type        TEXT NOT NULL,
  confidence  TEXT NOT NULL DEFAULT 'certain',
  metadata    TEXT NOT NULL DEFAULT '{}',
  updated_at  INTEGER NOT NULL
);

CREATE TABLE analysis_meta (
  key         TEXT PRIMARY KEY,
  value       TEXT NOT NULL
);
-- 存储：last_analyzed_at, project_root, framework_detected, codeomnivis_version

CREATE INDEX idx_nodes_type     ON nodes(type);
CREATE INDEX idx_nodes_file     ON nodes(file_path);
CREATE INDEX idx_edges_source   ON edges(source);
CREATE INDEX idx_edges_target   ON edges(target);
CREATE INDEX idx_edges_type     ON edges(type);
```

### 6.6 模块聚合规则

当节点总数超过 **100 个**时，自动进行聚合：

1. **API 路由聚合**：按路由前缀第一段分组（`/booking/**` → `booking` 模块节点）
2. **组件聚合**：按目录路径分组（`components/booking/` → `Booking UI` 模块节点）
3. **DB 模型不聚合**：DB 层节点始终完整显示（通常 < 50 个）
4. **聚合阈值**：单个模块内节点 > 5 个才触发折叠

```typescript
interface ModuleNode extends OmniNode {
  type: 'module'
  metadata: {
    childCount: number
    childTypes: NodeType[]
    routePrefix?: string   // 对于 API 路由模块
    dirPath?: string       // 对于组件模块
  }
}
```

## 7. 解析引擎详细设计

### 7.1 解析流水线

```
文件系统扫描
     ↓
文件分类（前端/后端/schema）
     ↓
并行解析（每类文件独立 parser）
     ↓
符号解析（跨文件 import 追踪）
     ↓
图构建（节点去重、边生成）
     ↓
一致性检测
     ↓
写入 SQLite
```

### 7.2 文件分类规则

```typescript
function classifyFile(filePath: string, projectMeta: ProjectMeta): FileClass {
  if (filePath.endsWith('schema.prisma')) return 'prisma_schema'
  if (filePath.match(/\.entity\.(ts|js)$/)) return 'typeorm_entity'
  if (filePath.match(/app\/.*\/page\.(tsx|jsx|ts|js)$/)) return 'nextjs_page'
  if (filePath.match(/app\/.*\/route\.(ts|js)$/)) return 'nextjs_api_route'
  if (filePath.match(/pages\/api\//)) return 'nextjs_pages_api'
  if (filePath.match(/pages\/(?!api\/)/)) return 'nextjs_page_legacy'
  if (projectMeta.trpcRouterPaths.some(p => filePath.startsWith(p))) return 'trpc_router'
  if (filePath.match(/\.(tsx|jsx)$/) && !filePath.includes('__tests__')) return 'react_component'
  if (filePath.match(/\.(ts|js)$/) && projectMeta.backendDirs.some(d => filePath.startsWith(d))) return 'backend_module'
  return 'unknown'
}
```

### 7.3 Prisma Schema 解析器

使用官方 `@prisma/internals` 的 `getDMMF` 函数，无需手写解析器。

```typescript
import { getDMMF } from '@prisma/internals'

async function parsePrismaSchema(schemaPath: string): Promise<OmniNode[]> {
  const dmmf = await getDMMF({ datamodelPath: schemaPath })
  const nodes: OmniNode[] = []

  for (const model of dmmf.datamodel.models) {
    // 生成 Model 节点
    nodes.push({
      id: `db_model:${schemaPath}:${model.name}`,
      type: 'db_model',
      name: model.name,
      filePath: schemaPath,
      line: 0, // Prisma DMMF 不暴露行号，需要用 regex 补充
      column: 0,
      metadata: {
        tableName: model.dbName ?? model.name,
        fieldCount: model.fields.length,
        fields: model.fields.map(f => ({
          name: f.name,
          type: f.type,
          isRequired: f.isRequired,
          isId: f.isId,
          isRelation: f.relationName !== undefined
        }))
      }
    })

    // 生成关系边
    for (const field of model.fields) {
      if (field.relationName && field.type) {
        edges.push({
          id: `${model.name}--db_relation--${field.type}:${field.relationName}`,
          source: `db_model:${schemaPath}:${model.name}`,
          target: `db_model:${schemaPath}:${field.type}`,
          type: 'db_relation',
          confidence: 'certain',
          metadata: {
            relationType: field.isList ? 'one_to_many' : 'one_to_one',
            fieldName: field.name,
            relationName: field.relationName
          }
        })
      }
    }
  }

  return nodes
}
```

**行号补全策略**：使用 `grep` 在原始 schema 文件中按 model 名称定位行号，作为后处理步骤。

### 7.4 Next.js App Router 解析器

```typescript
import * as path from 'path'
import { glob } from 'fast-glob'

function extractNextjsRoutes(appDir: string): OmniNode[] {
  const pageFiles = glob.sync('**/page.{tsx,jsx,ts,js}', { cwd: appDir })
  const routeFiles = glob.sync('**/route.{ts,js}', { cwd: appDir })
  const nodes: OmniNode[] = []

  for (const file of pageFiles) {
    const route = filePathToRoute(file, appDir)
    nodes.push({
      id: `page:${file}:${route}`,
      type: 'page',
      name: route || '/',
      filePath: file,
      line: 1, column: 1,
      metadata: {
        route,
        isDynamic: route.includes('['),
        params: extractRouteParams(route),
        isGroupLayout: route.includes('('),
        layoutFile: findLayoutFile(file, appDir)
      }
    })
  }

  for (const file of routeFiles) {
    const route = filePathToRoute(file, appDir)
    const methods = extractHttpMethods(file)  // 解析 export function GET/POST/PUT/DELETE
    for (const method of methods) {
      nodes.push({
        id: `api_route:${file}:${method}:${route}`,
        type: 'api_route',
        name: `${method} ${route}`,
        filePath: file,
        line: method.line, column: 1,
        metadata: { method: method.verb, route, isNextApiRoute: true }
      })
    }
  }

  return nodes
}

// /app/booking/[id]/page.tsx → /booking/[id]
function filePathToRoute(filePath: string, appDir: string): string {
  return '/' + filePath
    .replace(appDir, '')
    .replace(/\/(page|route|layout)\.(tsx|jsx|ts|js)$/, '')
    .replace(/^\//,'')
    .replace(/\(.*?\)\//g, '')  // 移除 route groups
```

### 7.5 tRPC Router 解析器

tRPC 的路由定义是 TypeScript 代码，需要使用 `ts-morph` 进行 AST 分析。

```typescript
import { Project, SyntaxKind, CallExpression } from 'ts-morph'

function parseTrpcRouter(routerFile: string, tsConfigPath: string): OmniNode[] {
  const project = new Project({ tsConfigFilePath: tsConfigPath })
  const sourceFile = project.addSourceFileAtPath(routerFile)
  const nodes: OmniNode[] = []

  // 找到所有 createTRPCRouter({ ... }) 调用
  sourceFile.getDescendantsOfKind(SyntaxKind.CallExpression).forEach(callExpr => {
    const expr = callExpr.getExpression().getText()
    if (!expr.includes('createTRPCRouter') && !expr.includes('router')) return

    const args = callExpr.getArguments()
    if (args.length === 0) return

    const routerDef = args[0]
    if (routerDef.getKind() !== SyntaxKind.ObjectLiteralExpression) return

    // 提取每个 procedure
    routerDef.getProperties().forEach(prop => {
      const procedureName = prop.getName()
      const procedureCall = prop.getFirstDescendantByKind(SyntaxKind.CallExpression)
      if (!procedureCall) return

      const procType = inferProcedureType(procedureCall)  // query / mutation / subscription
      const routerName = inferRouterName(routerFile)       // 从文件路径推断

      nodes.push({
        id: `trpc_procedure:${routerFile}:${routerName}.${procedureName}`,
        type: 'trpc_procedure',
        name: `${routerName}.${procedureName}`,
        filePath: routerFile,
        line: prop.getStartLineNumber(),
        column: prop.getStart(),
        metadata: {
          procedureType: procType,
          routerName,
          procedureName,
          hasInput: hasInputSchema(procedureCall),
          hasOutput: hasOutputSchema(procedureCall)
        }
      })
    })
  })

  return nodes
}

function inferProcedureType(call: CallExpression): 'query' | 'mutation' | 'subscription' {
  const chainText = call.getFullText()
  if (chainText.includes('.query(')) return 'query'
  if (chainText.includes('.mutation(')) return 'mutation'
  if (chainText.includes('.subscription(')) return 'subscription'
  return 'query'  // 默认
}
```

### 7.8 跨文件符号追踪（Resolver Layer）

这是解析引擎最核心也最复杂的部分。

**追踪目标**：给定 API route handler 函数，找到它调用的所有 DB 操作。

```typescript
import { Project, SourceFile } from 'ts-morph'

class SymbolResolver {
  private project: Project
  private visited = new Set<string>()  // 防止循环追踪

  // 从 handler 函数追踪到 DB 操作
  traceHandlerToDb(handlerFn: FunctionDeclaration | ArrowFunction): DbCall[] {
    const calls = this.extractCallExpressions(handlerFn)
    const dbCalls: DbCall[] = []

    for (const call of calls) {
      if (this.isPrismaCall(call)) {
        dbCalls.push(this.extractPrismaCall(call))
        continue
      }
      // 追踪到被调用的函数定义
      const definition = this.resolveCalleeDefinition(call)
      if (definition && !this.visited.has(definition.id)) {
        this.visited.add(definition.id)
        dbCalls.push(...this.traceHandlerToDb(definition))
      }
    }
    return dbCalls
  }

  // 识别 Prisma 调用模式：prisma.user.findMany() 等
  private isPrismaCall(call: CallExpression): boolean {
    const text = call.getExpression().getText()
    return /prisma\.\w+\.(findMany|findFirst|findUnique|create|update|delete|upsert|count|aggregate)/.test(text)
  }

  private extractPrismaCall(call: CallExpression): DbCall {
    const text = call.getExpression().getText()
    const match = text.match(/prisma\.(\w+)\.(\w+)/)
    return {
      modelName: match?.[1] ?? 'unknown',
      operation: match?.[2] ?? 'unknown',
      line: call.getStartLineNumber()
    }
  }
}
```

**追踪深度限制**：默认最多追踪 5 层调用，防止性能爆炸。可通过配置覆盖。

## 11. 前后端接口一致性检测

### 11.1 检测规则

**规则 1：死链 API 调用（Critical）**
- 前端 `fetch('/api/xxx')` 或 `axios.get('/api/xxx')`，但后端找不到对应路由定义
- 判断为 `confidence: 'certain'` 时才报 critical，`inferred` 时报 warning

**规则 2：未使用的 API 路由（Warning）**
- 后端定义了路由，但前端代码中没有任何调用
- 可能是冗余接口或遗留代码

**规则 3：HTTP Method 不匹配（Critical）**
- 前端 `POST /api/booking`，但后端定义了 `GET /api/booking`

**规则 4：tRPC 调用的 procedure 不存在（Critical）**
- 前端 `trpc.booking.delete.useMutation()` 但 booking router 中没有 `delete` procedure

**规则 5：动态路由参数不一致（Warning）**
- 前端调用 `/api/booking/${id}/cancel`，后端只定义了 `/api/booking/[id]`（缺少 `/cancel`）

### 11.2 检测算法

```typescript
function detectInconsistencies(graph: OmniGraph): Issue[] {
  const issues: Issue[] = []

  // 找到所有前端 API 调用边（type: 'calls_api'）
  const apiCalls = graph.edges.filter(e => e.type === 'calls_api')

  for (const call of apiCalls) {
    const callerNode = graph.getNode(call.source)
    const targetNode = graph.getNode(call.target)

    // 如果目标节点不存在 → 死链
    if (!targetNode) {
      issues.push({
        severity: call.confidence === 'certain' ? 'critical' : 'warning',
        type: 'dead_api_call',
        edge: call,
        caller: callerNode
      })
    }
    // 如果目标存在但 method 不匹配 → method 不一致
    else if (targetNode.metadata.method && call.metadata.method &&
             targetNode.metadata.method !== call.metadata.method) {
      issues.push({
        severity: 'critical',
        type: 'method_mismatch',
        edge: call,
        caller: callerNode,
        target: targetNode
      })
    }
  }

  // 找到所有后端路由节点
  const backendRoutes = graph.nodes.filter(n =>
    n.type === 'api_route' || n.type === 'trpc_procedure'
  )

  for (const route of backendRoutes) {
    const hasCallers = graph.edges.some(e => e.target === route.id && e.type === 'calls_api')
    if (!hasCallers) {
      issues.push({
        severity: 'warning',
        type: 'unused_route',
        target: route
      })
    }
  }

  return issues
}
```

### 12.2 每阶段的技术风险

**Phase 2 最高风险**：tRPC router 可能有嵌套 router（`appRouter` 包含 `bookingRouter` 包含 `create`）。需要递归解析 router 合并树。

**Phase 4 最高风险**：跨文件符号追踪可能遇到 cal.com 的复杂 monorepo 包边界，`ts-morph` 需要正确配置 `tsconfig.json` 路径。

**Phase 5 最高风险**：Cytoscape.js dagre 布局在几百节点时可能需要 > 1 秒，需要 Web Worker 执行布局计算。

## 15. 风险与缓解

| 风险 | 概率 | 严重度 | 缓解方案 |
|------|------|--------|---------|
| cal.com tRPC nested router 解析失败 | 高 | 高 | Phase 2 优先针对 cal.com 做测试，不通过不进入 Phase 3 |
| ts-morph 在 monorepo 中跨包追踪失败 | 中 | 高 | 降级为"同包内追踪"，跨包边标记 confidence='inferred' |
| Cytoscape 大图布局性能超时 | 中 | 中 | Phase 5 使用 Web Worker，超时后降级为 grid 布局 |
| cal.com 代码频繁更新导致 demo 失效 | 中 | 中 | 固定 demo 用的 cal.com commit hash，README 标注版本 |
| path alias 解析不完整 | 高 | 低 | 解析失败时标记 confidence='inferred'，不崩溃 |
| 竞品在发布前抢先推出类似工具 | 低 | 中 | 加快 Phase 1-4，尽早发布 beta |
| 大型项目分析超过 60 秒 | 中 | 中 | 增量缓存（SQLite），二次运行秒级完成 |

### 15.1 降级策略

所有解析器必须遵循**降级而非崩溃**原则：

```typescript
// 正确：无法解析时返回空数组，标记 warning
async function parseTrpcRouter(file: string): Promise<ParseResult> {
  try {
    return { nodes: await doParseRouter(file), errors: [] }
  } catch (err) {
    return {
      nodes: [],
      errors: [{ file, message: `tRPC parser failed: ${err.message}`, severity: 'warning' }]
    }
  }
}

// 错误：抛出异常中断整个分析
async function parseTrpcRouter(file: string): Promise<OmniNode[]> {
  return doParseRouter(file)  // ← 可能抛异常，不允许
}
```
